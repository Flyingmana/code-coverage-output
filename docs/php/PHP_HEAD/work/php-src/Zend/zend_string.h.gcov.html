     371             : /*
     372             :  * DJBX33A (Daniel J. Bernstein, Times 33 with Addition)
     373             :  *
     374             :  * This is Daniel J. Bernstein's popular `times 33' hash function as
     375             :  * posted by him years ago on comp.lang.c. It basically uses a function
     376             :  * like ``hash(i) = hash(i-1) * 33 + str[i]''. This is one of the best
     377             :  * known hash functions for strings. Because it is both computed very
     378             :  * fast and distributes very well.
     379             :  *
     380             :  * The magic of number 33, i.e. why it works better than many other
     381             :  * constants, prime or not, has never been adequately explained by
     382             :  * anyone. So I try an explanation: if one experimentally tests all
     383             :  * multipliers between 1 and 256 (as RSE did now) one detects that even
     384             :  * numbers are not usable at all. The remaining 128 odd numbers
     385             :  * (except for the number 1) work more or less all equally well. They
     386             :  * all distribute in an acceptable way and this way fill a hash table
     387             :  * with an average percent of approx. 86%.
     388             :  *
     389             :  * If one compares the Chi^2 values of the variants, the number 33 not
     390             :  * even has the best value. But the number 33 and a few other equally
     391             :  * good numbers like 17, 31, 63, 127 and 129 have nevertheless a great
     392             :  * advantage to the remaining numbers in the large set of possible
     393             :  * multipliers: their multiply operation can be replaced by a faster
     394             :  * operation based on just one shift plus either a single addition
     395             :  * or subtraction operation. And because a hash function has to both
     396             :  * distribute good _and_ has to be very fast to compute, those few
     397             :  * numbers should be preferred and seems to be the reason why Daniel J.
     398             :  * Bernstein also preferred it.
     399             :  *
     400             :  *
     401             :  *                  -- Ralf S. Engelschall <rse@engelschall.com>
     402             :  */
